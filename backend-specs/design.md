Blog Engine - design.md

This document outlines the technical design, architecture, and data models for the blog engine backend, based on requirements.md.

1. Architecture Overview

We will implement a Hexagonal (Ports and Adapters) architecture combined with Domain-Driven Design (DDD).

Core (Inside): domain (pure models, ports) and application (use case services).

Outside (Adapters):

Driving Adapters: resources/web (Unified Spring Controller handling JSON and HTML).

Driven Adapters: resources/persistence (PostgreSQL/JPA) and resources/security (JWT).

4. Application Layer Design

4.1. Application Services (Use Cases)

Spring @Service classes (e.g., PostService) that depend on the Domain Port interfaces (PostRepository, etc.).

They contain all the business logic (e.g., createPost, getPublishedPostBySlug).

4.2. Data Transfer Objects (DTOs)

All API DTOs (e.g., CreatePostRequest, PostResponse) will be auto-generated from the openapi.yml file's schemas section using the openapi-generator plugin.

5. Resources: Web Adapter (Unified Controller)

The Web Adapter will implement the API contract for JSON and provide HTML for HTMX.

5.1. Auto-Generated API Interface

The openapi-generator will create a Spring API interface (e.g., PostsApi) from the openapi.yml paths.

5.2. Unified Controller Implementation

Our controller will be annotated with @Controller (not @RestController).

It will implement the auto-generated PostsApi interface, ensuring contract compliance.

It will use the produces attribute to serve both JSON and HTML from the same URL.

// in resources/web/PostController.kt
@Controller
@RequestMapping("/posts")
class PostController(
private val postService: PostService
) : PostsApi { // <-- Implements auto-generated contract

    /**
     * JSON implementation for React/Mobile (from OpenAPI)
     * Note: @ResponseBody is needed here because the class is @Controller
     */
    @ResponseBody
    override fun getPostAsJson(slug: String): ResponseEntity<PostResponse> {
        val post = postService.getPublishedPostBySlug(slug)
        // ... logic to map and return ResponseEntity
    }

    /**
     * HTMX implementation for Web
     * This method is NOT part of the OpenAPI spec.
     */
    @GetMapping("/{slug}", produces = [MediaType.TEXT_HTML_VALUE])
    fun getPostAsHtmx(@PathVariable slug: String, model: Model): String {
        val post = postService.getPublishedPostBySlug(slug)
        model.addAttribute("post", post)
        return "fragments/post-detail" // Returns Thymeleaf template name
    }
}


6. Resources: Persistence Adapter (JPA/Hibernate)

JPA Entities: @Entity classes (e.g., PostEntity) in resources/persistence/model/.

Mappers: Code to convert between Post (domain) and PostEntity (persistence).

JPA Repositories: Standard JpaRepository interfaces.

Port Adapters: @Component classes that implement the domain ports (e.g., PostRepositoryAdapter).

7. Resources: Security Adapter (JWT)

Standard Spring Security setup with UserDetailsService (using UserRepository port), a JwtTokenProvider, and a JwtAuthenticationFilter.

SecurityConfig will protect /api/admin/* paths.

8. API Contract (openapi.yml)

The openapi.yml file will be the single source of truth for the JSON API. It will define all paths (/api/auth/login, /api/admin/posts, etc.) and all schemas (CreatePostRequest, PostResponse, etc.).
